## 自动化实现说明（程序如何对应本流程）

> 本节说明上述手动流程在 `web-register-sub2api-automation` 里的自动化实现，以及本次两处修复。

- **情况 1（图1-2，重新登录 + 填码）= 已修复并自动**：程序检测到「欢迎回来 + 邮箱输入框」即判为情况 1 → 自动填邮箱 → 点「继续」→ **稳健等待**「检查你的收件箱」码页（最长 ~30s，邮箱框还在就**自动重点一次「继续」**）→ 调邮箱验证码服务取码 → **自动填入「验证码」** → 点「继续」。
  - 🔧 修复（两点）：① **主因**——授权页「重新登录」的验证码邮件主题是「你的临时 **OpenAI** 登录代码」，而取码规则曾固定只匹配主题含 `ChatGPT`，导致**项目A 已收到邮件、项目B 却取不到码**（408 超时），页面随后被空提交——这正是你测到的「验证码没自动填、然后点了继续」。已改为主题关键词 `ChatGPT,OpenAI`（项目 A 支持逗号 OR）。② **加固**——浏览器侧旧逻辑码框 15s 没出现就静默跳过填码；现改为等待码页+必要时重点继续+**取到码必填**，取不到则**显式失败**（不会空提交），邮箱保留 `registered` 可续传。
- **情况 2（图3，选择账号）= 自动**：无邮箱框、有账号卡片 → 自动点击当前账号卡片。情况 1 与情况 2 互斥。
- **电话验证（图4-5）= 自动（接码成功率受平台限制）**：国家切到与接码一致 → 5sim 下单 → 填号（去区号）→ 优先 Text Message，进不去下一步则切 WhatsApp → 取短信码填「代码」→ 继续；「号码已被验证过」等→取消订单、返回、换号重试（`SMS_MAX_PHONE_ATTEMPTS`）。
- **回调（图6/图8）= 自动**：程序监听网络抓 `/auth/callback?code=&state=`（**无需手动 F12**）→ **只认与本流程 state 一致的回调**（忽略同一 Chrome 里其它/手动打开的授权流，避免 state mismatch）→ 调 Sub2API `exchange-code` + 建号绑 `self` 分组。
- **图6「使用 ChatGPT 登录到 Codex」后的两种情况 = 自动**：点继续后——情况B 直接出回调 → 建号（最常见）；**情况A 弹出图7「查看你的手机」第二道验证**（发往别的号码、收不到的码）→ 程序识别 `_is_check_phone_page` → **不去填**，而是用 Sub2API **重新生成授权链接重开页面**（`OAUTH_MAX_ATTEMPTS` 默认 5），重走 选账号/重新邮箱登录 → 回调。
- **手机环节失败的自愈 = 自动**：某个号收不到码导致流程回退到「登录或注册」起点时，程序识别 `_is_oauth_start_page` → 同样**重开新授权链接**重试（换新号），直到成功或用尽次数；第二次又弹手机号输入页也走重开。

> 🔧 另一处修复（邮箱被「烧掉」）：旧逻辑**注册成功就把邮箱标记 used**，导入若失败邮箱永久不可用。现在**注册成功只标 `registered`，导入成功才标 `used`**；`registered`（已注册待导入）可用 `POST /api/imports/resume` **续传**——自动重新走上面的情况 1（重新邮箱登录接码）补做 Sub2API 导入。
